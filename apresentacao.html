Propósito Educacional & Social

O Sistema AEE destina-se a gerenciar com total segurança os registros do Atendimento Educacional Especializado (AEE). Ele atua como um facilitador pedagógico para organizar o Plano de Desenvolvimento Individualizado (PDI) e relatórios especializados de alunos com Necessidades Educacionais Especiais (NEE) em escolas.

Privacidade por Design (LGPD)

Dado que laudos e diagnósticos de menores são considerados dados pessoais sensíveis, o sistema oculta a visualização por padrão na interface. Para que um professor autorizado consulte dados confidenciais, ele deve clicar explicitamente em "Revelar", o que gera um log imutável de auditoria no backend de forma imediata.

Decisões Críticas de Arquitetura
Estratégia Local-First com Reação Síncrona

As salas de recursos escolares das redes públicas sofrem com sinal de rede muito oscilante. Se a aplicação dependesse diretamente de conexões HTTP constantes, os professores teriam o fluxo de trabalho interrompido de forma recorrente. Adotamos o IndexedDB local como **banco de dados primário e única fonte de verdade** da UI para garantir resiliência absoluta.

IndexedDB Reativo vs LocalStorage Convencional IndexedDB

Em vez de persistência simples em LocalStorage (bloqueante, limitada a **5MB** e restrita a cadeias de texto plano), o WebAEE opera de forma assíncrona, não-bloqueante e transacional através do IndexedDB. Isso nos permite armazenar **grandes arquivos binários de fotos pedagógicas** diretamente como dados de blob locais para posterior envio.

Camada de Domínio & Modelos Locais

Pureza do Domínio na Clean Architecture

Na arquitetura do frontend, a camada de **Domínio** isola completamente as regras de negócio puras e modelos de dados do ecossistema de rede e endpoints HTTP. Ela define os contratos estruturais das tabelas locais do navegador, regras de validação pré-cadastro e a lógica de priorização transacional offline.

Fila de Sincronização Transacional com Precedência

Para otimizar conexões lentas em salas de recursos, a fila local de sync (`sync_queue`) classifica operações em prioridades estritas: **Prioridade 1** para relatórios e diários (críticos e extremamente leves em banda), e **Prioridade 2** para dados de alunos e mídias binárias pesadas (enviados sequencialmente em segundo plano).

Evidências do Esquema IndexedDB e Contratos de Tipo do Domínio
Versionamento e Schema IndexedDB (Dexie) frontend/lib/db.ts 
class AeeDatabase extends Dexie {
  alunos!: Table<AlunoLocal>
  relatorios!: Table<RelatorioLocal>
  fotos!: Table<FotoLocal>
  sync_queue!: Table<SyncQueueItem>

  constructor() {
    super('aee_db')
    this.version(1).stores({
      alunos:     '++id, server_id, nome, status, sync_status, updated_at',
      relatorios: '++id, server_id, aluno_id, tipo, sync_status, updated_at',
      fotos:      '++id, server_id, aluno_id, tag_pedagogica, sync_status, created_at',
      sync_queue: '++id, entidade, prioridade, criado_em',
    })
  }
}

Camada de Aplicação & Hooks Reativos

Casos de Uso Customizados e Reativos

A camada de **Aplicação** orquestra o fluxo de dados, implementando os Casos de Uso pedagógicos em hooks reativos customizados. O hook `useAlunos` extrai do banco local aplicando filtros e buscas de forma isolada, enquanto o `useSync` gerencia a drenagem em lote e o tratamento de re-tentativa transparente na fila transacional.

Controle Reativo Local com useLiveQuery

Substituímos técnicas ultrapassadas de consulta periódica (*polling* por intervalos) pela reatividade nativa do `useLiveQuery` da biblioteca `dexie-react-hooks`. A interface assina as tabelas do IndexedDB e re-renderiza componentes de forma instantânea quando ocorre persistência local ou sincronizações automáticas em segundo plano.

Evidência Prática (Escrita de Serviço & Leitura de Hook Customizado)
Serviço de Persistência Local (Escrita) frontend/lib/services/alunoLocalService.ts 
export async function salvarAlunoLocal(
  dados: Omit<AlunoLocal, 'id' | 'sync_status' | 'updated_at'>
): Promise<number> {
  const agora = new Date().toISOString()

  // Persiste no IndexedDB local com status pendente
  const id = await db.alunos.add({
    ...dados,
    sync_status: 'pending',
    updated_at: agora,
  })

  // Enfileira o sync (aplica prioridade 2 para Alunos)
  await enqueue('aluno', 'create', { ...dados, local_id: id })

  return id as number
}
Hook Customizado Reativo (Leitura) frontend/hooks/useAlunos.ts 
export function useAlunos(
  filtroStatus?: 'ativo' | 'arquivado',
  filtroBusca?: string
) {
  // useLiveQuery reage imediatamente a alterações no DB
  const alunos = useLiveQuery(
    () => filtroStatus
        ? db.alunos.where('status').equals(filtroStatus).toArray()
        : db.alunos.toArray(),
    [filtroStatus]
  )

  const filtrados = useMemo(() => {
    if (!alunos) return []
    return alunos.filter((a) => 
      !filtroBusca || a.nome.toLowerCase().includes(filtroBusca.toLowerCase())
    )
  }, [alunos, filtroBusca])

  return { alunos: filtrados, loading: alunos === undefined }
}

Infraestrutura HTTP & Proteção LGPD

Centralização de Rede e Barreira de Acesso

A camada de **Infraestrutura** implementa a interface concreta com a API backend FastAPI do Sistema AEE. Proibimos chamadas brutas de `fetch` nas views, direcionando todo o tráfego pelo `apiClient` encapsulado, o que centraliza a injeção do token JWT Bearer e o controle de segurança global.

Intercepção Crítica 401 e Auditoria PG

Para garantir total conformidade com a LGPD em máquinas públicas escolares compartilhadas, se a chamada à API falhar com **401 Unauthorized** (sessão expirada), o cliente destrói imediatamente o token local e os dados sensíveis do aluno carregados e redireciona para `/login`. A leitura de laudos é encriptada e registra logs de auditoria imutáveis no PostgreSQL.

Evidências de Infraestrutura: Cliente HTTP e Consulta de Auditoria (pgAdmin)
Interceptor 401 & JWT automatizado frontend/lib/api/client.ts 
async function request<T>(path: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('aee_token')
  const res = await fetch(`${API_BASE}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { 'Authorization': `Bearer ${token}` } : {}),
      ...options.headers,
    },
  })

  // Expiração de Sessão: limpa credenciais imediatamente
  if (res.status === 401) {
    localStorage.removeItem('aee_token')
    window.location.href = '/login'
    throw new ApiError(401, 'Sessão expirada.')
  }
  if (!res.ok) throw new ApiError(res.status, 'Erro na API')
  return res.json()
}

Garantia de Qualidade & Testes BDD

Especificação de Testes no Padrão BDD

A robustez das operações de sincronização offline e persistência local é assegurada por testes unitários e de integração escritos no **Vitest** sob as diretrizes rigorosas do padrão **BDD (Given / When / Then)**. Isso assegura que novas atualizações não gerem quebras ou regressões nas regras transacionais.

Mocking Estruturado e Isolamento de Loop

O hook assíncrono `useLiveQuery` depende do IndexedDB real e de transações assíncronas do browser. Para simplificar a execução e isolar a fila de sincronização pedagógica local nos testes, mockamos o `useLiveQuery` de forma **síncrona** nos arquivos de especificação técnica, permitindo asserções instantâneas.
